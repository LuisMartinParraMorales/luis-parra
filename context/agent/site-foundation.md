# Site Foundation

## Purpose

This repository is for Luis's public-facing portfolio and personal website.

The site should help with:

- recruiter and hiring-manager screening
- portfolio credibility
- interview preparation
- job applications across several closely related role families

It should not feel like a generic personal site or a polished stack list.

## Core Positioning

Primary positioning statement:

Luis is a software and ML engineer who turns transport, geospatial, and simulation workflows into dependable decision-support products.

Supporting ideas:

- prototype-to-product engineering
- backend-heavy analytical systems
- graph and generalized-cost methods
- cloud-aware runtime and orchestration thinking
- technically serious about validation, parity, and maintainability

## Site Objectives

The site should accomplish five things quickly:

1. Explain what Luis builds in one clear sentence.
2. Show that his work sits at the intersection of algorithms, software delivery, and decision support.
3. Give inspectable proof points, not generic capability claims.
4. Make transport, geospatial, and simulation depth feel like an advantage rather than a niche.
5. Support applications to Software & ML Engineer, ML Systems Engineer, and Data Scientist roles with strong software depth.

## Audience

Primary audiences:

- recruiters
- hiring managers
- technical interviewers
- collaborators who want to understand Luis's work quickly

Secondary audiences:

- transport and geospatial practitioners
- people reviewing public code or project writeups

## Project Story Rules

Each project should follow the same structure:

1. Problem
2. Constraints
3. What I built
4. Hard decisions
5. Outcome

This keeps the site focused on judgment and mechanisms rather than tool lists.

## Public Content Rules

### Do

- keep claims evidence-based
- use concrete mechanisms and decisions
- show technical depth through story, not jargon
- name what Luis actually built or changed
- prefer "this is the part I owned" over inflated ownership language

### Avoid

- generic "passionate about AI" copy
- vague skill inventories
- claims that depend on private deployment details
- architecture descriptions with no personal contribution signal
- public stories that read like sanitized changelogs

## Public-Safety Rules

Before publishing any project story:

- remove internal paths, URLs, and account details
- generalize confidential client context when it is not needed
- keep engineering patterns and decision logic
- be explicit when a story is about architecture contribution rather than primary implementation ownership

## Visual Strategy

Preferred visuals:

- workflow diagrams
- architecture diagrams
- maps
- ranked tables
- validation or before/after comparisons
- selected code or contract screenshots where public-safe

Avoid:

- decorative filler graphics
- stock-photo hero sections
- charts with no explanatory role in the story

## Role Strategy

The site should support:

- Software & ML Engineer roles
- ML Systems / AI Systems Engineer roles
- Data Scientist roles where software and algorithmic depth matter

The site should therefore make all of these visible:

- backend services, runtimes, contracts, tests, orchestration
- algorithms, graph methods, transport and geospatial reasoning
- full-stack analytical interfaces where Luis owned meaningful product shape

## Writing Style

The public site should sound:

- clear
- technically literate
- specific
- grounded
- quietly ambitious

It should not sound:

- over-marketed
- vague
- timid
- buzzword-heavy
