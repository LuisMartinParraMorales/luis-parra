# Projects Roadmap

This file lists the project stories we should turn into public-facing case studies.

## Priority order

### 1. Argo / `gtfs_skims`

Why first:

- strongest public proof point
- clear algorithmic signal
- easy to link directly to code
- supports software, ML-systems, and data-science applications

Public angle:

- open-source GTFS skimming and generalized-cost graph workflows built for scale and correctness

## 2. Polaris

Why second:

- clearest prototype-to-product story
- best combination of backend, frontend, contracts, and testing
- strong "analytical workflow turned into usable product" narrative

Public angle:

- evolving a workbook-driven prioritisation prototype into a cleaner internal decision-support platform

## 3. NTAP Journey Time Runtime

Why third:

- strongest runtime, contract, and integration story
- good evidence for backend-heavy and ML-systems-adjacent roles
- shows validation, execution flow, and platform-facing engineering

Public angle:

- building journey-time runtime and validation components inside a larger transport analytics platform

## 4. Basildon / VAMOS

Why fourth:

- strong simulation and post-processing story
- good evidence for AWS Batch, parquet outputs, intervention comparison, and analyst-facing outputs
- distinctive transport / ABM domain signal

Public angle:

- making agent-based simulation outputs operational through reusable post-processing, map, and comparison workflows

## 5. CRDT

Why fifth:

- strong architecture and systems-thinking signal
- useful for digital-twin, resilience, and public-sector positioning
- needs more careful ownership wording than the projects above

Public angle:

- architecture and platform-design work for a crisis-response and resilience analytics system

## Rule for all project stories

Every project page should answer these five questions clearly:

1. What was the problem?
2. What were the constraints?
3. What did I build?
4. What hard decisions did I make?
5. What changed as a result?

## Story rule

When a project mixes architecture, product, and analysis, prefer the story that shows how the technical pieces fit together rather than listing tools. The point is to show engineering judgment, not stack accumulation.
