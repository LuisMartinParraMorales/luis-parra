# Project Intake Checklist

Use this file to collect the missing project information Luis should provide before we turn internal notes into strong public case studies.

The goal is not to gather everything possible. The goal is to gather the minimum information that makes each project page:

- technically specific
- publicly safe
- visually compelling
- honest about Luis's contribution

## Core Information Needed For Every Project

For each project, I need the following:

### 1. Public-safe title and summary

- project name we should use publicly
- one-sentence summary in plain English
- whether the client or organization name can be mentioned directly

### 2. Real problem

- what real decision, workflow, or operational pain this project addressed
- who the users were
- why the old workflow was not enough

### 3. Constraints

- what made the problem hard
- what data, performance, policy, delivery, or organizational constraints mattered
- what had to be preserved from a previous workflow or prototype

### 4. Luis's ownership

- what Luis directly built
- what Luis co-built
- what Luis reviewed, supported, or influenced but should not overclaim
- which parts are strongest to present as evidence of personal capability

### 5. System and implementation shape

- main backend, frontend, ML, algorithmic, and cloud pieces
- whether the system was batch, API, event-driven, scheduled, or interactive
- what technical decisions Luis is comfortable defending in interviews

### 6. Hard decisions and tradeoffs

- the two or three tradeoffs that mattered most
- what was intentionally deferred
- what choice improved correctness, speed, maintainability, or usability

### 7. Outcome

- measurable outcomes if they are safe to share
- if hard metrics are not available, operational outcomes we can describe honestly
- what changed for users, analysts, or the team

### 8. Public links and references

- repo link if public
- demo link if public
- docs, screenshots, or diagrams that are safe to publish

### 9. Visual assets

For each project, I ideally want:

- 1 hero image
- 1 system or workflow diagram
- 1 product or output screenshot
- 1 supporting image showing maps, charts, tables, or data output

### 10. Confidentiality boundary

- what details must stay private
- what should be generalized
- whether there are screenshots we can crop, blur, anonymize, or recreate

## Visual Types That Work Best

These are the strongest image types for your portfolio:

- product UI screenshots
- map screenshots
- workflow diagrams
- architecture diagrams
- ranked tables
- charts with a clear caption
- before/after comparisons
- code screenshots only when they show something unusually clear or elegant

Avoid these unless they add real signal:

- stock photos
- generic laptop mockups
- decorative AI imagery
- charts without explanation
