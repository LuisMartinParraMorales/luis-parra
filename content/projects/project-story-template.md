# Project Story Template

Use this structure for every public-facing project page.

## Title

Project name

## One-line summary

A short sentence that explains the project in plain language.

## Problem

- What real-world problem existed?
- What workflow or decision needed support?
- Why was the existing approach not enough?

## Constraints

- What made the problem hard?
- What data, operational, stakeholder, or technical limits shaped the solution?
- What had to be preserved from the prior workflow or prototype?

## What I Built

- What system, workflow, feature set, or architecture did I create or improve?
- What were the main technical components?
- What was the shape of the solution from the user’s point of view?

## Hard Decisions

- What tradeoffs mattered most?
- What did I intentionally not build yet?
- What choices improved correctness, maintainability, or usability?

## Outcome

- What improved?
- What became faster, clearer, more reproducible, or easier to use?
- If precise metrics are not available, describe the operational value honestly.

## Technical Highlights

- list the most relevant implementation details
- keep this selective, not exhaustive

## What This Project Says About Me

- What capability does this project prove?
- Why should a hiring manager care?

## Visual Ideas

- architecture diagram
- workflow diagram
- map
- chart
- table
- before/after comparison

## Public-Safety Check

Before publishing:

- remove confidential details
- generalize sensitive project context where needed
- keep the engineering story, remove unnecessary internal specifics
